Aim

Objective

This experiment aims to provide a thorough understanding of TCP Congestion Control mechanisms — the suite of algorithms that prevent network congestion collapse while maintaining efficient utilisation of available bandwidth.

Learning Outcomes

Upon completion of this experiment, students will be able to:

  • Describe the role of the Congestion Window (cwnd) and its dynamic adjustment.
  • Explain the Slow Start phase and its exponential growth behaviour.
  • Distinguish between Slow Start and Congestion Avoidance (linear growth).
  • Demonstrate Fast Retransmit triggered by three duplicate ACKs.
  • Trace the Fast Recovery procedure and contrast it with timeout-based recovery.
  • Interpret cwnd-versus-time graphs produced by TCP Reno.
Lab Domain: Computer Networks  |  Estimated Time: 45–60 minutes  |  Level: Undergraduate / Graduate

Theory

1. Congestion and the Need for Control

Network congestion occurs when the offered load exceeds a link's capacity, leading to packet queuing, delays, and ultimately packet loss. TCP detects congestion implicitly through packet loss events (timeout or duplicate ACKs) and adjusts its sending rate accordingly.

2. Congestion Window (cwnd)

The congestion window (cwnd) is a sender-side limit on the number of unacknowledged bytes (segments) in flight. The effective window is:

EffWin = min(cwnd, rwnd)

where rwnd is the receiver-advertised window.

3. Slow Start

TCP begins with cwnd = 1 MSS. For every ACK received, cwnd increases by 1 MSS — effectively doubling each RTT (exponential growth). This continues until cwnd ≥ ssthresh or until packet loss is detected.

  • Start: cwnd = 1
  • On each ACK: cwnd += 1
  • Doubles per RTT until threshold

4. Congestion Avoidance

Once cwnd ≥ ssthresh, TCP switches to linear (additive) growth — AIMD (Additive Increase, Multiplicative Decrease):

cwnd += MSS × (MSS / cwnd) ≈ +1 MSS per RTT

5. Fast Retransmit

When the sender receives three duplicate ACKs for the same segment, it infers that the segment was lost (but later segments arrived). TCP retransmits the missing segment immediately — without waiting for a timeout. This is far faster than a retransmission timeout (RTO).

6. Fast Recovery (TCP Reno)

After Fast Retransmit, TCP Reno enters Fast Recovery:

  • ssthresh = cwnd / 2
  • cwnd = ssthresh + 3 (inflate by 3 duplicate ACKs)
  • For each additional duplicate ACK: cwnd += 1
  • On new ACK: cwnd = ssthresh, resume Congestion Avoidance

Procedure

Follow the steps below to perform the TCP Congestion Control simulation experiment:

  1. Navigate to the Simulation tab using the left sidebar.
  2. Set the initial ssthresh value using the input field provided. The default value is 16 segments. This is the threshold at which TCP transitions from Slow Start to Congestion Avoidance.
  3. Click "Start" to begin the simulation. Observe the congestion window (cwnd) growing exponentially during the Slow Start phase (doubling each RTT).
  4. Observe the phase indicator on screen. When cwnd reaches ssthresh, the phase changes to Congestion Avoidance; growth slows to approximately +1 MSS/RTT.
  5. Click "Introduce Packet Loss" at any point during the simulation. This simulates detection of 3 duplicate ACKs. The simulation will:
    • Set ssthresh = cwnd / 2
    • Transition to Fast Recovery
    • Reduce cwnd to the new ssthresh
    • Resume Congestion Avoidance
  6. Observe the graph: note the sawtooth pattern typical of TCP Reno — gradual increase followed by abrupt halving on congestion.
  7. Use "Pause" at any time to freeze the simulation and examine the current state values.
  8. Click "Reset" to return to the initial state (cwnd = 1).
  9. Repeat with different ssthresh values (e.g., 8, 32) and observe how the transition point changes.
  10. Proceed to Posttest after completing at least two simulation runs.
Note: Each step on the graph represents one Round-Trip Time (RTT). The simulation is accelerated for demonstration purposes.
